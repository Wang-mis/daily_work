# 状态码
- 1xx —— 接收的请求正在处理
	- 101 Switch Protocols 切换协议，如WebSocket协议升级
- 2xx —— 请求已正常处理完毕
	- 200 OK
	- 201 Created 服务器已创建相应资源
	- 204 No Content 请求处理成功，响应中不包含正文（常用于DELETE）
	- 206 Partial Content 范围请求成功（常用于断点续传/分块下载）
- 3xx —— 重定向
	- 301 Moved Permanently 永久重定向
	- 302 Found 临时重定向
	- 304 Not Modified 协商缓存命中
- 4xx —— 客户端错误
	- 400 Bad Request
	- 401 Unauthorized
	- 403 Forbidden
	- 404 Not Found
	- 405 Method Not Allowed
- 5xx —— 服务器错误
	- 500 Internal Server Error
	- 502 Bad Gateway
	- 503 Server Unavailable
	- 504 Gateway Timeout

> **301 / 302**
> - 301和302都是重定向状态码，浏览器在接收到301或302状态码后，会跳转到响应头中的Location所指向的网址。
> - 两者的区别在于SEO（搜索引擎优化），当搜索引擎遇到301状态码时，会将原网址的权重（索引、排名等信息）转移到新的网址。当遇到302时，不会转移权重，而是保留原网址的权重。

# 方法
- GET
	- 从服务器中获取资源
	- 安全、幂等
	- 参数通过URL传递
- POST
	- 提交数据到服务器
	- 不安全、非幂等
	- 参数通过请求体传递
- PUT
	- 修改服务器上的资源
- PATCH
	- 修改服务器上的资源的部分内容
- DELETE
	- 删除服务器上的资源
- HEAD
	- 与GET一致，但没有响应体
	- 用法
		- 判断服务器上是否有某资源
		- 下载资源前获取元数据（资源类型、大小等）
		- 缓存校验
- OPTIONS
	- [[CORS#预检请求|预检请求]]
	- 用法
		- 询问服务器对当前URL可使用哪些请求方法
		- [[CORS]] 检查服务器允许的域、请求方法、请求头
- TRACE
	- 回显
# 缓存
HTTP有两种缓存机制：强制缓存和协商缓存。
- 强制缓存：发送请求前由浏览器自行判断缓存是否失效，如果未失效，则直接从缓存中读取，不发送请求。
- 协商缓存：如果强制缓存失效，则浏览器与服务器协商判断当前缓存是否失效。

## 强制缓存
与强制缓存有关的请求头：
- `Expires` HTTP 1.0使用的强制缓存请求头，值为绝对时间。
- `Cache-Control` HTTP 1.1开始使用，使用相对时间。可用取值：
	- `max-age=60` 表示缓存在60秒后失效
	- `no-cache` 不使用强制缓存
	- `no-store` 不使用缓存（包括强制缓存和协商缓存）
	- `private`
	- `public`

> `private`和`public`的作用是区分谁可以缓存。
> 在一次网络请求和响应的过程中，不只有浏览器有缓存能力，中间经过的CDN、代理服务器等都具有缓存功能。
> - `private` 只有用户浏览器可以缓存该请求
> - `public`  任何环节均可缓存该请求

## 协商缓存
1. 第一次请求，服务器发送的响应中包含`LastModified/ETag`字段，浏览器将该字段存储起来
2. 下一次请求同样的url时，如果强制缓存失效，就会在请求头中添加`If-Modified-Since/If-None-Match`字段，值为之前储存的`LastModified/ETag`字段的值。
3. 服务器根据该字段判断是否需要更新缓存
	- 如果需要，则发送200 OK并在响应体中发送更新后的数据
	- 如果不需要，则发送304 Not Modified

# HTTP版本更新
## HTTP 1.1
- **长连接**
	- `Connection: keep-alive`，使用一个TCP连接处理多个HTTP请求，减少TCP连接建立与关闭的时间消耗。
- **管道化**
	- 运行在一个TCP连接上并行处理多个HTTP请求。但服务器必须按顺序返回，造成了**队头阻塞**问题。
- **缓存控制**
	- 引入了`Cache-Control, ETag, Last-Modified`缓存机制。
- **分块传输**
## HTTP 2
- **二进制分帧**
	- 在HTTP和TCP之间引入了二进制分帧层，将消息分成多个帧。
	- 消息头部被分成`HEADERS`帧，如果头部太大超过上限，则会分成一个`HEADERS`帧和多个`CONTINUATION`帧。
	- 数据部分可以被分成0个或多个`DATA`帧。
- **多路复用**
	- 通过分帧技术，HTTP2可以做到多个消息并行处理、交错传输，解决了HTTP层面的队头阻塞问题。
- **头部压缩**
	- 在HTTP通信过程中，大部分头部都不会发生变化，但是因为HTTP是无状态协议，每次请求和响应都需要传输这些头部，造成了带宽的浪费，为了解决这个问题，HTTP 2引入了HPACK头部压缩算法。
	- 对于一个TCP连接，客户端和服务器各自维护两张表：**静态索引表**和**动态索引表**
		- 对于常见的头部字段，直接传输它在静态索引表中的索引号
		- 对于自定义头部，第一次传输（经过huffman编码后压缩的）完整内容，并将这个头部存入动态索引表，之后的消息直接使用索引号
- **服务端推送**
	- 对于客户端发起的一个请求，服务器可以发送多个响应。当客户端发起了对html文件的请求后，服务器推测客户端也需要html文件内引用的css、js等资源，于是直接向客户端推送这些资源。
	- 问题
		- 缓存竞争：服务器很难知道浏览器缓存了哪些资源，如果主动推送的资源之前就已经被缓存过，再推送一次反而浪费了带宽
		- 优先级冲突：服务器主动推送资源，可能抢占高优先级资源的带宽
	- 替代方案 `HTTP 103 Early Hints`
## HTTP 3
为了避免**TCP层面的队头阻塞**问题，HTTP3不再使用TCP作为传输层协议，而是使用了**基于UDP**的**QUIC协议**。
- QUIC协议默认集成TLS 1.3，将传输握手与TLS握手合并，实现1-RTT或0-RTT的建立连接时延。
- 使用QPACK算法替代HTTP 2的HPACK算法。
- 在用户态实现拥塞控制，不依赖于操作系统内核更新。
- 使用`Connection ID`标识连接，不依赖于IP地址，从而实现IP迁移后的无缝继续。
